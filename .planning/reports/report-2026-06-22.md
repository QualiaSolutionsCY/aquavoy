## What Was Done
- **Investigated a live production complaint** — Wency's chat "wasn't answering like he wanted." Went through his actual logged-in session in the browser + the production traces, and root-caused it to FOUR separate layered bugs (the visible symptom hid the real ones).
- **Fixed file reads — URL-encoded OneDrive names.** `read_file`'s `extractText` sniffed type with `name.endsWith(".pdf")`, but OneDrive returns the name percent-encoded (`Fam%20J%2Epdf`), so every PDF/Word/Excel fell to the "unknown binary" branch and returned nothing. Decode the name first (PR #18, + regression test).
- **Fixed the agent going silent.** When the tool loop exhausted its budget the agent returned "(no response)". Added, in escalating order: a system nudge (#19), a forced non-streaming answer (#20), and finally a **clean-prompt fallback** (#21) — re-ask with the question + a digest of tool results, which even gemini-flash answers. No more "(no response)".
- **Fixed PDFs being unreadable in serverless (the big one).** Captured the exact error live: `DOMMatrix is not defined`. `pdf-parse` v2 uses pdf.js's *browser* build, which needs the `DOMMatrix` DOM global absent from Vercel's Node runtime — so **every PDF** failed (it only looked like "scanned/unreadable files"). Swapped to **`unpdf`** (serverless-native pdf.js with polyfills baked in); removed `pdf-parse` (PR #22). Validated live: unpdf extracts PDF text in Node with no DOMMatrix error.
- **Set the agent model.** Tested claude-sonnet-4.6 (works great) but reverted to **`google/gemini-3.5-flash`** per cost preference, and made flash reliable via the clean-prompt fallback above. Model now lives in the `OPENROUTER_MODEL` Vercel env (claude is a one-var swap if reliability ever beats cost).
- **Shipped + verified** all of it in Wency's live session — flights return real web-search results; the employee/file lookup now answers honestly instead of rambling or going silent.
